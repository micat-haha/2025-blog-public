最近用 Vibe Coding 做了一个小项目：一个临时文件中转站。

项目地址：  
https://github.com/micat-haha/temp-file-transfer

最开始做它的原因很简单：我只是想要一个方便传文件的小工具。平时手机、电脑、平板之间经常需要临时传一些文件，用微信、QQ 或网盘当然也可以，但总觉得不够直接。我想要的是一个更轻量的东西：打开网页，上传文件，生成链接或者取件码，另一台设备直接下载，过几天自动删除。

所以这个项目一开始的定位就很简单：它不是云盘，也不打算做成复杂的文件管理系统，只是一个临时文件中转站。

## 从一个很小的需求开始

刚开始的时候，我其实没有想太多。只是觉得“上传文件—生成链接—下载文件”这个流程应该不难。

但真正开始做之后才发现，一个看起来很小的需求，只要想让它真正可用，就会不断冒出新的问题。

比如：

文件上传后怎么找回来？  
别人怎么通过取件码下载？  
文件过期后怎么自动删除？  
上传错了能不能自己删？  
后台能不能看到现在有哪些文件？  
部署到服务器之后会不会被别人乱用？  
如果要上传大文件，普通用户和自己用的时候要不要区分权限？

这些问题不是一开始就想清楚的，而是在开发过程中一点点出现的。

这次让我感受很深的一点是：很多需求不是提前设计出来的，而是做着做着长出来的。

## 先做出能跑的版本

第一版我没有追求完整，只想先把最基本的流程跑通。

能上传文件，能生成下载链接，能生成取件码，能下载，能设置保存时间，到期能自动清理。做到这一步的时候，其实已经挺有成就感了。因为它已经从一个脑子里的想法，变成了一个真的可以打开、可以上传、可以下载的网页。

但这种成就感很快就会被各种报错打断。

有时候页面打不开，有时候上传成功了但结果页报错，有时候文件已经保存了但前端显示失败，有时候本地环境缺少某个依赖。每次看到终端里一大串报错，第一反应都是：又坏了。

不过也正是在这些报错里，我慢慢进入了一种新的开发节奏：

发现问题，复制报错，描述现象，让 AI 修改，然后重新运行，再测试。

它不一定每次都对，但它一直能给出下一步方案。而我需要做的是判断这些方案是否合理，哪里需要调整，哪里不能直接照做。

## 功能一点点变多

项目最开始只是一个简单的上传下载页面。

后来我觉得，如果文件多了，至少需要一个后台看看现在有哪些文件。于是加了管理员后台，可以查看文件、搜索文件、筛选状态、删除文件、清理过期文件。

再后来，我发现用户上传完文件之后，如果刷新页面，很容易找不到刚刚上传的东西。于是又加了当前设备的上传历史，可以看到文件名、取件码、上传时间、下载次数和状态。

之后我又觉得，删除文件不应该单独生成一个删除链接。删除链接容易丢，也不够自然。于是改成了在上传成功后和历史记录里直接显示删除按钮，点击后确认删除。

再往后，我开始考虑部署到服务器之后的情况。如果完全开放上传，别人可能会乱传大文件。所以我又加了普通上传限制和隐藏高级上传码。普通用户可以直接上传小文件，自己或朋友知道隐藏入口和高级码，就可以上传更大的文件。

做到这里的时候，我也明显感觉到，项目开始变大了。

一个功能接着一个功能加上去，很容易让项目失控。所以我后来一直提醒自己：这个项目只是临时文件中转站，不是云盘。它不需要用户系统，不需要文件夹，不需要长期存储，也不需要多端同步。

保持边界很重要。

## 我不断改变自己的想法

这次开发过程中，我的很多想法都变过。

一开始我想做删除链接，后来觉得不好用，于是改成删除按钮。

一开始我觉得后台不是必须的，后来发现没有后台很难管理。

一开始我只想本地跑一下，后来又开始考虑 GitHub、Docker、Nginx 和服务器部署。

一开始我觉得功能越多越好，后来又觉得功能越多越容易偏离最初需求。

这让我意识到，做项目时很难一开始就把所有事情想清楚。更真实的过程是：先做出一个能跑的东西，然后在使用和测试中不断发现哪里不顺手，再一点点改。

这和我平时写科研脚本也有点像。很多时候不是先有一个完美方案，而是先跑起来，看结果，再修改，再验证。

## AI 提高的是迭代速度

这次最大的感受不是“AI 替我写了代码”，而是它明显加快了我的迭代速度。

以前做一个这样的小网站，很多时间会花在项目结构、接口、模板、数据库、前端交互、部署配置这些细节上。每个点都不算难，但加起来很耗时间。

现在我可以先用自然语言把需求描述出来，让 AI 生成第一版，然后自己运行、测试、发现问题，再继续调整。

但这并不意味着我可以完全不用思考。

我仍然要判断：

这个功能有没有必要？  
这个设计会不会太复杂？  
这个接口会不会有安全问题？  
普通用户能不能看到不该看到的信息？  
上传失败会不会残留文件？  
Docker 挂载会不会导致数据丢失？  
GitHub 上会不会误传数据库和上传文件？

AI 能帮我更快地写代码，但方向、取舍和验证还是要自己来。

## 从能跑到能用

这个项目让我明显感受到，“能跑”和“能用”之间其实差了很多细节。

能上传文件只是第一步。真正想让它好用，还要考虑很多小地方：

上传成功后有没有清楚的反馈；  
下载链接和取件码能不能一键复制；  
手机端页面会不会太挤；  
二维码扫码是否方便；  
删除前有没有确认；  
历史记录能不能找回之前上传的文件；  
后台能不能看到异常文件；  
部署后文件和数据库会不会丢；  
普通用户会不会滥用上传入口。

这些细节都不大，但它们决定了这个项目是不是真的能用。

## 做完之后的感觉

第一版做完之后，我有一种挺踏实的感觉。

它不是一个很大的项目，也没有多复杂的架构，但它是完整的。它有上传下载，有取件码，有历史记录，有后台，有权限限制，有 Docker 部署方式，也能继续往服务器上部署。

更重要的是，它解决的是我自己的一个真实需求。

我很喜欢这种感觉：不是为了做一个很宏大的系统，而是把生活里一个小小的不方便，变成一个自己能控制、能修改、能部署的小工具。

## 一点感悟

这次做下来，我觉得小项目最重要的不是一开始就设计得多完美，而是先让它跑起来。

先做一个最小版本。  
然后不断测试。  
不断修问题。  
不断删掉不必要的复杂想法。  
始终记得它最初是为了解决什么问题。

这个临时文件中转站一开始只是一个很小的想法，但因为它足够具体，所以很容易开始；也因为它确实有用，所以我愿意继续完善它。

最后它变成了一个我自己会用，也愿意继续维护的小工具。

对我来说，这就是这次开发最有价值的地方。
